## 2024-05-14 - Intermediate Directory Permissions Vulnerability & Shared Path Architecture
**Vulnerability:** A vulnerability was identified during the execution of `STORAGE_DIR.mkdir(mode=0o700, parents=True, exist_ok=True)`. The `.mkdir()` function in `pathlib` only applies the specified mode to the *last* (leaf) directory created. If intermediate directories were also implicitly created, they would inherit the system's default umask permissions (often overly permissive like `0o755` or `0o777`), potentially exposing sensitive files in the application's configuration path structure.

**Learning:** Initial attempts to fix this by iterating over all implicitly created intermediate directories and aggressively restricting them to `0o700` proved to be an architectural risk. If an application is the first to create standard shared paths (like `~/.config/`, `~/.local/share/`, or `/opt/`), restricting those shared parent directories to `0o700` breaks access for other applications and services that rely on them.

**Prevention:** To safely handle deep path creation without breaking shared system architectures, the `parents=True` parameter should be used without specifying a restrictive mode during `mkdir`. This allows intermediate directories to be created cleanly with safe default permissions. Then, apply `chmod(0o700)` *only* to the application-specific leaf directory (`STORAGE_DIR`) after creation. This ensures sensitive data is protected while maintaining the integrity of shared directory structures. Tests verifying directory creation security (`tests/test_security.py`) should explicitly assert that intermediate directories are NOT restricted to `0o700`.
